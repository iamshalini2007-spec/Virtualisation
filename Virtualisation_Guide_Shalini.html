Virtualisation — The Big Picture
// What it is, why it exists, and the two dominant platforms
ℹ DEFINITION

Virtualisation is the process of creating a software-based (virtual) version of a physical resource — a server, storage device, or network — so multiple virtual machines (VMs) can run on a single physical host simultaneously, each isolated from the others.

Before virtualisation, one application ran on one server. A web server used 5% CPU. A database used 8%. Both sat idle 90% of the time. Virtualisation fixes this — run 20 VMs on one physical server, each using their slice of resources.

💡 KEY INSIGHT

Both Hyper-V (Microsoft) and VMware ESXi (Broadcom) are Type 1 hypervisors. They run directly on bare metal — there is no host OS underneath them. This is why they're fast and suitable for production workloads.

Key Terminology — Universal
VM — Virtual Machine

A software-defined computer with its own vCPU, vRAM, vDisk, vNIC. Completely isolated. Can run any OS. Portable — can be moved between hosts.

vCPU — Virtual CPU

A logical CPU assigned to a VM. Multiple vCPUs can map to the same physical core (overcommit). A VM with 4 vCPUs doesn't need 4 dedicated physical cores.

Snapshot

A point-in-time capture of a VM's disk + memory state. Roll back instantly. NOT a backup — snapshots degrade performance over time and must be deleted after use.

Live Migration / vMotion

Moving a running VM from one host to another with zero downtime. Memory is copied in iterations while VM runs; final cutover takes milliseconds.

Overcommit

Assigning more vCPUs or vRAM to VMs than physical resources exist. Works because most VMs don't use 100% at the same time. Risky if all VMs spike simultaneously.

Host / Guest

Host = the physical server running the hypervisor. Guest = the VM running ON the host. A host runs many guests.

The Partition Model
⚡ CRITICAL CONCEPT — Unique to Hyper-V

Hyper-V uses partitions, not processes. The Parent Partition (root) is the management OS — it has privileged access. Child Partitions are VMs — they have NO direct hardware access. All hardware requests from VMs go through the parent partition via VMBus.

Parent Partition (Root Partition)

The Windows Server OS where you installed the Hyper-V role. Hosts the Virtualization Service Providers (VSPs). Has direct hardware driver access. Runs the Hyper-V Manager and PowerShell management. NOT a VM — it's the management layer.

Child Partition (VM)

Each VM is a child partition. Uses synthetic devices (VMBus) for fast I/O to parent. Has its own virtual address space, vCPUs, vRAM. Cannot access hardware directly — everything goes through hypercalls to Ring -1.

VMBus — The Communication Channel

A high-speed, memory-mapped communication channel between parent and child partitions. Replaces slow emulated devices with synthetic ones. Requires Integration Services (Linux: linux-vm-tools) installed in the guest. Without VMBus: VM falls back to slow emulated IDE/network.

Key Hyper-V Components
► vmms.exe — Virtual Machine Management Service▾

The core Windows service that manages VM lifecycle. Handles Create/Start/Stop/Delete VM. Communicates with Hyper-V hypervisor via WMI (Msvm_ namespace). If vmms.exe stops, all VMs still run but you lose management capability.

Get-Service vmms | Select Name,Status
Restart-Service vmms  # Does NOT affect running VMs
► Integration Services (IC)▾

Guest-side drivers that enable synthetic device support: Hyper-V Data Exchange, Heartbeat, Time Synchronization, Shutdown, VSS (backup). Without ICs: VM uses slow emulated devices.

VMkernel — The Core
ℹ VMkernel vs Parent Partition

Unlike Hyper-V which uses a privileged parent partition (Windows), VMware ESXi's VMkernel runs directly on hardware with NO host OS. It's a purpose-built microkernel. Management is done via a separate management plane (hostd/vpxa daemons) not a full OS.

VMM — Virtual Machine Monitor

Each VM gets its own VMM process. The VMM intercepts privileged instructions from the guest and translates them. Modern VMMs use Intel VT-x/AMD-V for hardware-assisted virtualisation — the CPU itself handles privilege trapping.

hostd — Host Daemon

The main management daemon on ESXi. Handles all vSphere API calls (SOAP/REST). Manages VM lifecycle, datastore operations, network config. If hostd crashes, VMs keep running but management is lost. Restart: /etc/init.d/hostd restart

vpxa — vCenter Agent

Installed on each ESXi host. Acts as the communication bridge between vCenter Server and the individual host's hostd. If vpxa is down, vCenter loses sight of the host but VMs continue running.

vmware-tools — Guest Integration

VMware Tools = Hyper-V Integration Services equivalent. Installs synthetic NIC driver (vmxnet3), SCSI driver (pvscsi), time sync, quiesced snapshots, graceful shutdown. Always install in every VM.

Hyper-V Virtual Networking
Hyper-V Virtual Switch (vSwitch)

A software Layer-2 switch inside Hyper-V. Connects VMs to each other and to external networks via physical NICs. Three types: External, Internal, Private.

Switch TypeVM-to-VMVM-to-HostVM-to-NetworkUse
External✓✓✓ (via pNIC)Production VMs needing network access
Internal✓✓✗Lab VMs + host management
Private✓✗✗Fully isolated VM-to-VM only
# Create external virtual switch bound to physical NIC
New-VMSwitch -Name "ExternalSwitch" -NetAdapterName "Ethernet" -AllowManagementOS $true

# Create internal switch
New-VMSwitch -Name "InternalSwitch" -SwitchType Internal

# Assign VM to switch
Get-VMNetworkAdapter -VMName "Server01" | Connect-VMNetworkAdapter -SwitchName "ExternalSwitch"

# VLAN tagging on VM NIC
Set-VMNetworkAdapterVlan -VMName "Server01" -Access -VlanId 100
VMware Virtual Networking
Standard vSwitch (vSS)

Host-local virtual switch. Configured per-host — not shared across hosts. Fine for small deployments. Max 1016 ports. Uplinks to physical NICs (vmnics).

Distributed vSwitch (dvSwitch / VDS)

Managed centrally from vCenter. Configuration applies to ALL hosts in the cluster simultaneously. Supports advanced features: LACP, traffic shaping per VM, NetFlow, port mirroring. Requires vCenter — cannot use with standalone ESXi.

Port Groups

Logical grouping of ports on a vSwitch with shared policy (VLAN, security, traffic shaping). VMs connect to port groups, not directly to vSwitches. Each port group can have different VLAN IDs.

VMkernel Adapters (vmk)

Special virtual NICs for the hypervisor itself — not for VMs. Used for: Management traffic (vmk0), vMotion, vSAN, iSCSI, NFS. Each vmk has its own IP and is bound to a port group.

# ESXi networking commands
esxcli network vswitch standard list
esxcli network vswitch standard portgroup list
esxcli network ip interface list
esxcli network ip address list

# Add VMkernel for vMotion
esxcli network ip interface add -i vmk1 -p "vMotion"

Storage & Virtual Disks
// Datastores, disk types, shared storage, and best practices
Hyper-V Storage
VHDX — Preferred Disk Format

64TB max. Supports 4K native sector disks. Resilient — metadata journal prevents corruption on power failure. Supports online resize and shared VHDX for clusters.

Shared VHDX

A .vhdx file shared between multiple VMs simultaneously — used for Guest Clusters (SQL FCIs inside VMs). Mounted as shared disk inside the guest cluster, not as individual VM disks.

Pass-through Disks

Bypass the virtualization layer — VM gets raw access to a physical disk or LUN. Best performance. Downside: disk can't be snapshotted or live-migrated without storage also migrating.

Storage Live Migration

Move a VM's .vhdx files to different storage while the VM is running. No downtime. Uses a mirror-and-cutover process. Takes longer than Live Migration but doesn't need identical storage on both hosts.

# Hyper-V storage operations
New-VHD -Path "D:\VMs\VM1.vhdx" -SizeBytes 200GB -Fixed
Add-VMHardDiskDrive -VMName "Server01" -Path "D:\VMs\VM1.vhdx"
Resize-VHD -Path "D:\VMs\VM1.vhdx" -SizeBytes 300GB  # Expand online
Move-VMStorage -VMName "Server01" -DestinationStoragePath "E:\VMs"
VMware Storage
Datastore (VMFS)

VMware File System — a cluster-aware filesystem allowing multiple ESXi hosts to read/write the same LUN simultaneously. All VM files (.vmdk, .vmx, .nvram) live in datastores. Versions: VMFS-5 (2TB LUNs), VMFS-6 (64TB LUNs, auto space reclaim).

NFS Datastore

Mount an NFS share as a datastore. Simpler than VMFS (no LUN management). Good for VMs that don't need maximum I/O. vSAN is preferred for performance-critical workloads.

vSAN

Software-defined storage. Pools local NVMe/SSD/HDD across cluster nodes into a shared datastore. Eliminates external SAN. Requires vCenter + minimum 3 hosts + SSDs. Each host contributes storage to the shared pool.

RDM — Raw Device Mapping

VMware's pass-through equivalent. Maps a physical LUN directly into a VM. Two modes: Virtual compatibility (snapshots work) and Physical compatibility (higher performance, no snapshots).

# VMware storage commands (PowerCLI)
Get-Datastore | Select Name, FreeSpaceGB, CapacityGB
Get-VM "Server01" | Get-HardDisk | Select Name, CapacityGB, StorageFormat
New-HardDisk -VM "Server01" -CapacityGB 200 -StorageFormat EagerZeroedThick
Move-HardDisk -HardDisk (Get-HardDisk -VM "Server01") -Datastore "FastSSD"

High Availability & Live Migration
// How VMs survive host failures and move without downtime
Hyper-V Live Migration
ℹ Live Migration Requirements

Both hosts: same CPU family (or compatible mode), same Hyper-V version, shared storage (SMB 3.0 or SAN) or Storage Migration, domain-joined, Live Migration enabled in Hyper-V settings, Kerberos or CredSSP delegation configured.

Live Migration Process
Phase 1

VM memory is copied from source to destination host while VM continues running (iterative copy)

Phase 2

Dirty pages (changed during copy) are tracked and re-copied iteratively until delta is small

Phase 3

VM is briefly paused (milliseconds), final memory pages copied, VM resumes on destination

Phase 4

Gratuitous ARP sent — network switches update MAC tables to new host's port

Phase 5

Source VM deleted. Network connections resume — total pause: under 1 second for typical workloads

# Hyper-V Live Migration
Move-VM -Name "Server01" -DestinationHost "HyperV-Node2"
Move-VM -Name "Server01" -DestinationHost "HyperV-Node2" -DestinationStoragePath "D:\VMs"

# Check live migration settings
Get-VMHost | Select *LiveMigration*
Enable-VMMigration -ComputerName HyperV-Node1
Hyper-V HA (Failover Clustering)
⚡ Hyper-V HA = Failover Clustering

Hyper-V doesn't have its own HA engine. It relies on Windows Server Failover Clustering (WSFC) to provide HA for VMs. When a host fails: WSFC detects it (via heartbeat), restarts VMs on surviving nodes. This is NOT automatic without clustering configured.

# Set VM restart priority in cluster
Get-ClusterResource | Where ResourceType -eq "Virtual Machine" | Select Name, State
(Get-ClusterGroup "Server01").Priority = 3000  # Higher = more important
VMware vMotion (Live Migration)
ℹ vMotion Requirements

Shared storage (VMFS/NFS/vSAN), dedicated VMkernel adapter tagged for vMotion, same virtual switch config on both hosts, compatible CPU (Enhanced vMotion Compatibility / EVC recommended), vCenter required.

# PowerCLI vMotion
Move-VM -VM "Server01" -Destination (Get-VMHost "esxi02")
Move-VM -VM "Server01" -Destination (Get-VMHost "esxi02") -Datastore (Get-Datastore "FastSSD")

💡 HA vs Live Migration Key Difference

Live Migration / vMotion = planned movement, zero downtime, VM stays running the whole time.
HA = unplanned recovery, VM restarts on another host after original host fails. There IS downtime (VM restart time, typically 30–120 seconds).

Security in Virtualisation
// Isolation, Secure Boot, vTPM, encryption, and hardening
VM Isolation — The Foundation
ℹ How VMs Are Isolated

Each VM runs in its own partition/VMM with a separate virtual address space. A VM cannot directly read another VM's memory. The hypervisor enforces this at the CPU ring level. VM escape (breaking out of a VM to affect the host) is extremely rare and treated as a critical CVE.

Hyper-V Security Features
Shielded VMs

Encrypts VM disk + state with BitLocker. VM can only run on approved Guarded Hosts (attested via Host Guardian Service). Protects against rogue admins and compromised fabric. Fabric admins cannot see VM data.

vTPM — Virtual TPM 2.0

Each Gen 2 VM can have a virtual TPM chip. Enables BitLocker inside the VM, Windows Hello, Credential Guard. Required for Windows 11 VMs. Stored encrypted in VM config — protected by HGS or local key protector.

Secure Boot (Gen 2)

Validates the bootloader signature before allowing boot. Prevents rootkit injection at boot time. Supports Microsoft UEFI CA and Linux UEFI CA templates. Can be customised per VM.

Network Isolation

Private vSwitch = VMs cannot reach host or external network. VLAN segmentation. Port ACLs via Extended Port ACL feature. SDN (Software Defined Networking) for microsegmentation in Azure Stack HCI.

# Enable vTPM on a VM
Set-VMKeyProtector -VMName "Server01" -NewLocalKeyProtector
Enable-VMTPM -VMName "Server01"

# Configure Secure Boot
Set-VMFirmware -VMName "Server01" -EnableSecureBoot On -SecureBootTemplate "MicrosoftWindows"

# Check security settings
Get-VM "Server01" | Get-VMFirmware | Select SecureBoot,SecureBootTemplate
Get-VMSecurity -VMName "Server01"
VMware Security Features
VM Encryption

vSphere 6.5+. Encrypts .vmdk files using AES-256. Key management via external KMS (KMIP) or vSphere Native Key Provider (vNKP). Encrypted VMs cannot be read even if disk file is stolen. Performance overhead ~5%.

vTPM (VMware)

Virtual TPM 2.0 for VMs. Requires VM Encryption and a key provider configured. Enables BitLocker, Windows Hello in Windows VMs. Stored as encrypted .nvram file.

NSX Micro-segmentation

Distributed firewall running inside the hypervisor kernel. Applies firewall rules at the vNIC level — even East-West traffic between VMs on the same host is filtered. Rules follow VMs when they vMotion. Zero-trust networking.

ESXi Hardening Guide

VMware publishes official hardening guides per ESXi version. Key items: disable SSH when not needed, restrict management network access, enable lockdown mode (restricts direct host access), use certificate-based auth.

# VMware security via PowerCLI
Get-VM "Server01" | Get-VMEncryptionInfo
Set-VMEncryption -VM "Server01" -Enable

# ESXi lockdown mode
Get-VMHost "esxi01" | Get-VMHostAuthentication | Set-VMHostAuthentication -LockdownMode Strict
Troubleshooting Guide
// Systematic diagnosis for common Hyper-V and VMware issues
ℹ First Steps for ANY Issue

1. Check event logs (Hyper-V: Event Viewer → Microsoft-Windows-Hyper-V-VMMS | VMware: /var/log/vmkernel.log)
2. Check host resource utilisation (CPU, RAM, storage I/O)
3. Verify host connectivity and network
4. Check VM tools status

Hyper-V Issues
⚠ VM fails to start — "Not enough memory"▾

Symptoms: VM won't power on. Error: "Not enough memory available to start VM"

Causes:

  • Host RAM over-committed — too many VMs running
  • VM startup RAM exceeds available host RAM
  • Dynamic Memory minimum set too high
  • Memory reserved for parent partition too low
Get-VMMemory -VMName "Server01"
(Get-VMHost).MemoryAvailable
Get-VM | Select Name, MemoryAssigned, MemoryDemand | Sort MemoryDemand -Desc

# Reduce startup RAM or enable Dynamic Memory
Set-VMMemory -VMName "Server01" -StartupBytes 2GB -DynamicMemoryEnabled $true -MinimumBytes 512MB
⚠ Live Migration fails — "Virtual machine migration failed"▾

Common causes and fixes:

  • CPU incompatibility: Enable "Migrate to physical computer with different processor version" in VM settings
  • Kerberos/CredSSP: Run migration from Hyper-V Manager (not PowerShell) or configure CredSSP delegation
  • Storage not accessible: Destination can't reach the VHDX path — use SMB share both hosts can access
  • Firewall blocking: Port 6600 (migration data) and 443 must be open between hosts
Test-NetConnection -ComputerName HyperV-Node2 -Port 6600
Get-VM "Server01" | Select *ProcessorCompatibility*
Set-VMProcessor -VMName "Server01" -CompatibilityForMigrationEnabled $true
⚠ VM network connectivity lost after migration▾
  • Virtual switch name mismatch between source and destination host
  • VLAN configuration missing on destination switch
  • Gratuitous ARP not propagated — physical switch ARP cache stale
Get-VMNetworkAdapter -VMName "Server01" | Select SwitchName, VlanSetting
Get-VMSwitch -ComputerName HyperV-Node2 | Select Name, SwitchType
# If switch name differs:
Connect-VMNetworkAdapter -VMName "Server01" -SwitchName "CorrectSwitchName"
⚠ Checkpoint (snapshot) causing disk performance issues▾
⚠ DANGER

Long-running checkpoints cause large .avhdx chains, consuming massive disk space and degrading I/O performance.

  • Delete all checkpoints when not needed
  • Never leave production VMs with checkpoints for more than hours
  • Merging a large checkpoint chain is slow and I/O intensive — plan a maintenance window
Get-VMSnapshot -VMName "Server01"
Remove-VMSnapshot -VMName "Server01" -Name "Before Patch" -IncludeAllChildSnapshots
# Force merge immediately:
(Get-VM "Server01").CheckpointFileLocation
⚠ Hyper-V host BSOD / vmms.exe crash▾

If the HOST crashes (BSOD), all VMs on it stop abruptly (dirty shutdown). Actions:

  • Check Windows Event Log → System for BugCheck events
  • Analyse memory dump: WinDbg → !analyze -v
  • Common causes: driver incompatibility with hypervisor, faulty RAM, storage driver bug
  • After reboot: VMs auto-start if configured, run chkdsk on VHDX: Repair-VHD
Repair-VHD -Path "D:\VMs\Server01.vhdx" -All
Get-WinEvent -LogName System | Where {$_.Id -in 41,1001} | Select -First 10
VMware Issues
⚠ vMotion fails — "A general system error occurred"▾
  • No shared storage: Both hosts must see the same datastore — verify with esxcli storage filesystem list on both
  • vMotion VMkernel missing: Ensure vmkX is tagged for vMotion on both hosts
  • CPU compatibility: Enable EVC (Enhanced vMotion Compatibility) on the cluster in vCenter
  • Insufficient bandwidth: vMotion needs 10GbE recommended for large VMs
# Check vMotion VMkernel
esxcli network ip interface list | grep vmk
esxcli network ip interface tag get -i vmk1  # Should show: VMotion

# PowerCLI
Get-VMHost "esxi01" | Get-VMHostNetworkAdapter | Where {$_.VMotionEnabled}
⚠ VM shows "Invalid" or "Orphaned" in vCenter▾

Invalid: vCenter has a record but can't find the VM on the host. Orphaned: VM exists on host but vCenter lost track of it.

  • Right-click → "Remove from Inventory" (not delete) → re-register the .vmx file from the datastore browser
  • SSH to host: find /vmfs/volumes/ -name "*.vmx" | head -20
# Re-register from ESXi shell
vim-cmd solo/registervm /vmfs/volumes/DatastoreName/VMFolder/VM.vmx
⚠ ESXi host disconnects from vCenter▾

VMs keep running on disconnected host but can't be managed from vCenter.

  • Check vpxa service: /etc/init.d/vpxa status
  • Check network: ping vCenter IP from ESXi host's management IP
  • Check DNS: nslookup vCenter FQDN from ESXi
  • Certificate mismatch: re-connect host in vCenter and accept new certificate
# On ESXi host via SSH:
/etc/init.d/vpxa restart
/etc/init.d/hostd restart
ping <vCenter-IP>
esxcli network ip connection list | grep 443
⚠ VM disk space full — VMDK expansion▾
  • Expand VMDK in vCenter (VM must be powered OFF for most formats, or online for hot-add if OS supports it)
  • After expanding VMDK, extend partition inside guest OS (diskpart / growpart / lvextend)
  • With snapshots present: CANNOT expand — delete all snapshots first
# PowerCLI expand disk
Get-HardDisk -VM "Server01" | Set-HardDisk -CapacityGB 300

# Inside Windows guest after expansion:
# diskpart → select disk → select volume → extend

# Inside Linux guest:
growpart /dev/sda 2
resize2fs /dev/sda2  # ext4
xfs_growfs /        # xfs
⚠ High CPU Ready — VM performance degraded▾
⚡ CPU Ready Explained

CPU Ready = time the VM was ready to run but had to WAIT for a physical CPU to become available. High CPU Ready (>10%) = host is CPU over-committed. VMs are starving for CPU cycles.

  • Check esxtop → %RDY column — values above 10% indicate overcommit
  • Reduce vCPU count on non-critical VMs (fewer vCPUs = easier to schedule)
  • Migrate VMs to less busy hosts (DRS should handle this automatically)
  • Add physical CPUs to the host
# ESXi shell
esxtop  # then press 'c' for CPU view, sort by %RDY

# PowerCLI — get CPU stats
Get-VM | Get-Stat -Stat cpu.ready.summation -Realtime | Sort Value -Desc | Select -First 10
Common Scenarios & Guidance
// Real-world situations with step-by-step resolution
📋 Scenario 1: Migrate physical server to VM (P2V)▾

Goal: Convert a physical Windows server to a VM with minimal downtime.

⚡ Key Risk

HAL (Hardware Abstraction Layer) differences between physical and virtual hardware. Always test the VM before decommissioning the physical server.

1
Inventory the physical server
Document: OS version, disk layout, IP config, applications, services, dependencies. Run ipconfig /all, diskpart list disk, services.msc.
2
Choose P2V tool
Hyper-V: Disk2vhd (free, Microsoft). VMware: vCenter Converter Standalone (free). Both can convert while server is running (hot clone).
3
Create the VM shell
Create VM with same specs. For Hyper-V: Gen 1 if source is old BIOS system. For VMware: use recommended hardware version.
4
Run the conversion
Use Disk2vhd or Converter to clone. Schedule during low-usage period. Hot clone = no downtime during copy but final cutover needs brief stop.
5
Post-conversion steps
Boot VM, install Integration Services / VMware Tools, change IP (or keep same after physical is off), remove old physical drivers, activate OS.
6
Validation
Test all services, DNS resolution, application functionality. Keep physical server OFF but available for 2–4 weeks before decommission.
📋 Scenario 2: VM is consuming too much disk — snapshot sprawl▾

Symptoms: Datastore running out of space, VM performance degraded, backup jobs failing.

1
Identify snapshots
Hyper-V: Get-VMSnapshot -VMName * | Select VMName,Name,CreationTime,ParentSnapshotName
VMware: Get-VM | Get-Snapshot | Select VM,Name,Created,SizeGB | Sort SizeGB -Desc
2
Check for orphaned snapshots
VMware: delta vmdk files in datastore not listed in snapshot manager = orphaned. Browse datastore and look for -delta.vmdk files not associated with current snapshots.
3
Plan deletion window
Deleting large snapshots causes disk I/O spike during merge. Schedule during low-traffic window. Large checkpoints (100GB+) can take hours to merge.
4
Delete snapshots
VMware: "Delete All" in snapshot manager — merges all into base disk. Hyper-V: Remove-VMSnapshot with -IncludeAllChildSnapshots.
5
Prevent recurrence
Set backup retention policies. Never take manual snapshots without a tracking ticket. Configure vCenter alarms for snapshot age > 24 hours.
📋 Scenario 3: Host failure — recovering VMs quickly▾

Situation: ESXi host or Hyper-V host suddenly goes offline. VMs need to be recovered.

1
Confirm host is truly dead
Ping management IP, check iDRAC/iLO/IPMI console, check physical hardware LEDs. Distinguish: crashed vs network isolated vs powered off.
2
With HA configured (VMware)
vSphere HA automatically restarts VMs on surviving hosts within 30–120 seconds. Monitor in vCenter → Tasks. No manual action needed if admission control has capacity.
3
With Failover Clustering (Hyper-V)
WSFC detects node failure via heartbeat (5 missed = 5 seconds). Clustered VMs failover to surviving nodes automatically. Check: Get-ClusterGroup | Select Name,OwnerNode,State
4
Without HA/Clustering — manual recovery
Register VM from shared storage on another host. VMware: vim-cmd solo/registervm /vmfs/volumes/... Hyper-V: Import-VM from the VHDX location.
5
Post-recovery
Check VM disk consistency (dirty shutdown). Run fsck/chkdsk inside guest. Review application logs. Investigate root cause of host failure.
📋 Scenario 4: VM network not working after VLAN change▾
1
Verify VLAN ID on VM port
Hyper-V: Get-VMNetworkAdapterVlan -VMName "VM01" | Select OperationMode, AccessVlanId
VMware: Check port group VLAN ID in vCenter network configuration.
2
Verify physical switch trunk
The physical NIC connected to the vSwitch must be configured as a trunk port allowing the VLAN ID. Coordinate with network team if physical switch ports need updating.
3
Test with known-good VM on same VLAN
If another VM on the same port group/VLAN works, the issue is VM-specific (IP config, firewall). If no VMs on that VLAN work, it's a switch/port group issue.
4
Check guest IP configuration
Ensure VM has correct IP for the new VLAN subnet. VLAN change usually means IP change too.
📋 Scenario 5: Planning VM sizing — right-sizing guidance▾
💡 Common Mistake

Over-provisioning VMs wastes hypervisor resources. A VM with 16 vCPUs when it uses 2 is harder to schedule and increases CPU Ready times for ALL VMs on the host.

WorkloadvCPU StartvRAM StartDiskNotes
Web/App Server2–44–8GB60–100GBScale out (more VMs) not up
SQL Server (small)416–32GB200GB+Separate data/log/tempdb disks
Domain Controller24–8GB60GBNever oversize DCs
File Server24GBAs neededI/O bound — prioritise disk
Dev/Test VM1–22–4GB40–60GBDynamic Memory OK

Monitor actual usage for 2–4 weeks after deployment, then right-size. Use Performance Monitor (Windows) or vmstat/top (Linux) inside guests.
